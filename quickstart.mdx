---
title: "Get Started with Credith in Under 5 Minutes"
sidebarTitle: "Quickstart"

This guide takes you from zero to a fully running Credith instance on your local machine. By the end you will have the PostgreSQL database, the Express API, and the React frontend all running in Docker — plus demo data loaded and your first API call verified. The only thing you need to start is Docker and Node.js already installed.

## Steps

<Steps>
  <Step title="Prerequisites">
    Make sure the following tools are available on your machine before proceeding:

    | Tool | Minimum Version | Purpose |
    |------|-----------------|---------|
    | [Docker Desktop](https://www.docker.com/products/docker-desktop/) | Latest stable | Runs all three services (DB, API, frontend) |
    | Node.js | 18+ | Required to run `pnpm` CLI commands outside of Docker |
    | pnpm | 11+ | Package manager used by both `Service` and `Website` workspaces |

    Verify your setup:

    ```bash
    docker --version
    node --version
    pnpm --version
    ```
  </Step>

  <Step title="Clone the Repository">
    Clone the Credith monorepo and navigate into it:

    ```bash
    git clone https://github.com/RoyGeova07/Credith.git
    cd Credith
    ```

    The repository has two main workspaces:
    - `Service/` — Express REST API (Node.js + Sequelize)
    - `Website/` — React SPA (Vite)
  </Step>

  <Step title="Configure Environment Variables">
    The development Docker Compose file (`docker-compose.yml`) includes sensible defaults for local development. If you want to override any value, create a `.env` file in the project root or edit the `environment` block in `docker-compose.yml` directly.

    <Note>
      The `.gitignore` excludes `.env` files from version control. Never commit secrets to the repository.
    </Note>

    The full set of environment variables consumed by the `credith_service` container:

    | Variable | Default (dev) | Description |
    |----------|---------------|-------------|
    | `NODE_ENV` | `development` | Controls Swagger UI availability and logging |
    | `PORT` | `3000` | Port the Express server binds to |
    | `JWT_SECRET` | `visca_barca_visca_catalunya` | Secret used to sign JWT tokens |
    | `DB_HOST` | `credith_db` | PostgreSQL container hostname |
    | `DB_PORT` | `5432` | PostgreSQL port |
    | `DB_NAME` | `credith` | Database name |
    | `DB_USER` | `postgres` | Database user |
    | `DB_PASSWORD` | `123456` | Database password |
    | `DB_DIALECT` | `postgres` | Sequelize dialect |
    | `CORS_ORIGIN` | `http://localhost:4173` | Allowed frontend origin for CORS |

    The `credith_website` container takes one required build argument:

    | Variable | Default (dev) | Description |
    |----------|---------------|-------------|
    | `VITE_BASE_ROUTE` | `http://localhost:3000` | Base URL the frontend uses for all API requests |
  </Step>

  <Step title="Start Docker Compose">
    Spin up all three services — the database, the API, and the frontend — with a single command from the project root:

    ```bash
    docker compose up --build
    ```

    Docker Compose will:
    1. Pull `postgres:17` and start `credith_db` with a health check (`pg_isready`).
    2. Build the `Service/dockerfile` and start `credith_service` on port `3000` (waits for the DB to be healthy).
    3. Build the `Website/dockerfile` with `VITE_BASE_ROUTE=http://localhost:3000` and start `credith_website` on port `4173`.

    Wait until you see output similar to:

    ```
    credith_service  | Example app listening on port 3000
    ```
  </Step>

  <Step title="Run Migrations">
    Once the containers are running, apply all pending Sequelize migrations to create the `cd` schema and all tables. Run this inside the `credith_service` container:

    ```bash
    docker exec -it credith_service pnpm migrate
    ```

    Alternatively, if you have the `Service` dependencies installed locally and your `.env` is configured, you can run:

    ```bash
    cd Service
    pnpm migrate
    ```

    A successful run prints each migration file name followed by a confirmation message.
  </Step>

  <Step title="Seed Demo Data">
    Credith ships with two seeders:

    | Command | Seeder File | What it loads |
    |---------|-------------|---------------|
    | `pnpm seed:prd` | `20260516000000-prd-base-data.js` | Three roles (OWNER, ADMIN, EMPLOYEE), the base company *ServiCredith*, one store, one checkout machine, five product categories, and one admin user (`admin@credith.hn` / password `123456`) |
    | `pnpm seed:dev` | `20260521083057-mock-company-data.js` | Two demo companies, three stores, five users, six products, four categories, active CAIs and ranges, four clients, and 21 bills spanning January–June 2026 with full installment payment plan data |

    For a realistic local environment with charts and reports populated, run the dev seeder:

    ```bash
    docker exec -it credith_service pnpm seed:dev
    ```

    For a clean production-like baseline with just the essential roles and a single admin account:

    ```bash
    docker exec -it credith_service pnpm seed:prd
    ```
  </Step>

  <Step title="Open the App">
    Your Credith instance is now fully operational:

    | Service | URL | Description |
    |---------|-----|-------------|
    | React Frontend | [http://localhost:4173](http://localhost:4173) | Login, POS dashboard, reports, management panels |
    | Express API | [http://localhost:3000](http://localhost:3000) | Health check — returns `{ "message": "Hello from the backend!" }` |
    | Swagger UI | [http://localhost:3000/api-docs](http://localhost:3000/api-docs) | Interactive API reference (development mode only) |

    Log in with the seeded admin credentials:
    - **Email:** `admin@credith.hn`
    - **Password:** `123456`
  </Step>
</Steps>

## First API Calls

With the service running, verify it and create your first user:

<CodeGroup>

```bash Health Check
curl http://localhost:3000/
```

```bash Register a New User
curl -X POST http://localhost:3000/api/users \
  -H "Content-Type: application/json" \
  -d '{
    "firstName": "Jane",
    "secondName": "M.",
    "firstLastName": "Doe",
    "secondLastName": "Smith",
    "email": "jane@example.com",
    "password": "secret123"
  }'
```

</CodeGroup>

The health-check returns:

```json
{ "message": "Hello from the backend!" }
```

The `POST /api/users` endpoint registers the user, hashes the password with bcrypt, validates that the email is not already taken, and returns a signed JWT token you can use for authenticated requests (e.g., `PUT /api/users/update-password`).

<Note>
  The Swagger UI at [http://localhost:3000/api-docs](http://localhost:3000/api-docs) is only available when `NODE_ENV=development`. It documents every route in the API with request/response schemas, generated automatically from JSDoc annotations on each route file. This is the fastest way to explore the full API surface.
</Note>
